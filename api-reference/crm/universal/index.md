# Универсальные методы CRM: обзор методов и событий

{% note tip "" %}

Если вы разрабатываете интеграции для Битрикс24 с помощью AI-инструментов (Codex, Claude Code, Cursor), подключите [MCP-сервер](../../../sdk/mcp.md), чтобы ассистент использовал официальную REST-документацию.

{% endnote %}

Методы `crm.item.*` управляют объектами CRM: лидами, сделками, контактами, компаниями, счетами, предложениями и элементами смарт-процессов.

Единый интерфейс упрощает работу с разными объектами. Вместо отдельных команд для каждого объекта используйте универсальные методы с идентификатором типа `entityTypeId`.

> Быстрый переход: [все методы и события](#all-methods)

## Как начать работу

1. Определите тип объекта CRM `entityTypeId`. Значение возьмите из справочника [типов объектов CRM](../../data-types.md#object_type) или раздела [Смарт-процессы](./user-defined-object-types/index.md).
2. Получите список доступных полей для этого типа методом [crm.item.fields](./crm-item-fields.md).
3. Создайте новый элемент методом [crm.item.add](./crm-item-add.md) или получите список существующих методом [crm.item.list](./crm-item-list.md).
4. Получите данные конкретного элемента методом [crm.item.get](./crm-item-get.md).
5. Измените элемент методом [crm.item.update](./crm-item-update.md) или удалите методом [crm.item.delete](./crm-item-delete.md).

## Связи универсальных методов с другими объектами

**Тип объекта CRM.** Тип задается параметром `entityTypeId`. Он определяет структуру полей и логику работы метода.

**Объект CRM.** Конкретная запись идентифицируется парой `entityTypeId` и `id`. Эта связка используется в основных методах `crm.item.*`.

**Реквизиты.** Привязывайте реквизиты компаний и контактов методами [crm.requisite.link.*](../requisites/links/index.md).

**Родительские связи.** Элементы связываются между собой через поля `parentId` и `parentEntityTypeId`:

- `fields` возвращает информацию о полях родителей
- `get` отдает значения родительских полей
- `list` фильтрует, сортирует и добавляет в выборку значения полей родителей
- `add` и `update` поддерживают изменение значений этих полей

## Особенности имен полей

В базе данных поля хранятся в формате `UPPER_CASE`, а в REST используются имена в `camelCase`.

Пример: `ASSIGNED_BY_ID` превращается в `assignedById`.

Для пользовательских полей преобразование сложнее, потому что исходные имена часто содержат цифры и символы подчеркивания.

Стандартный случай: `UF_CRM_10_5186744711` превращается в `ufCrm10_5186744711`. Подчеркивания между числовыми блоками сохраняются, остальные убираются.

Проблема возникает, когда буквы и цифры идут вперемешку. Например, `UF_CRM_10_DIGIT10` преобразуется в `ufCrm10Digit10` При обратном преобразовании невозможно точно определить, было ли исходное поле `UF_CRM_10_DIGIT10` или `UF_CRM_10_DIGIT_10`.

С версии **CRM 21.1800.0** система проверяет такие конфликты перед преобразованием. Если имя становится неоднозначным, оно возвращается в исходном виде `UPPER_CASE`.

#|
|| **UPPER_CASE** | **camelCase** ||
|| `UF_CRM_3_DIGIT` | `ufCrm3Digit` ||
|| `UF_CRM_3_DIGIT10` | `ufCrm_3_DIGIT10` ||
|| `UF_CRM_3_DIGIT_10` | `ufCrm_3_DIGIT_10` ||
|| `UF_CRM_3_1747309727` | `ufCrm3_1747309727` ||
|| `UF_CRM_1747309879` | `ufCrm_1747309879` ||
|#

С версии **CRM 25.0.0** методы `crm.item.*` поддерживают параметр `useOriginalUfNames`, который управляет форматом имен пользовательских полей в запросе и ответе:

- `Y` — оригинальные имена пользовательских полей, например, `UF_CRM_2_1639669411830`
- `N` — имена пользовательских полей в `camelCase`, например, `ufCrm2_1639669411830`

По умолчанию используется `N`.
