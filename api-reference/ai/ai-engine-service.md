Вы можете создать собственный сервис для интеграции любой внешней модели AI с Битрикс24. Это позволяет использовать кастомные алгоритмы обработки текстов, аудио и изображений, не привязываясь к стандартным провайдерам.

Готовый упрощенный шаблон для создания такого сервиса [можно скачать по ссылке](https://helpdesk.bitrix24.ru/examples/endpoint.zip).

{% note warning %}

Готовый пример работает в один поток и не подходит под готовые коммерческие модели.
Используйте его исключительно в ознакомительных целях.

{% endnote %}

## Принцип работы сервиса

В вашем сервисе должно быть 2 потока выполнения:

- **Основной поток** (работа с сервисом):
    1. Прием запроса: скрипт принимает и валидирует входящие данные от Битрикс24.
    2. Постановка в очередь: запрос должен быть обработан максимально быстро, поэтому полученный запрос нужно записать во внутреннюю очередь выполнения.
    3. Возвращение успешного ответа: строго определенный формат json-сообщения.

- **Поток обработки** (асинхронное выполнение генерации):
    1. Получить запроса из очереди
    2. Обработать запрос при помощи языковой модели
    3. Отправить результат: специальный запрос на адрес, указанный в поле `callbackUrl`.

{% note alert %}

Ссылка `callbackUrl` имеет ограниченное время жизни (параметр `ttl`). Если вы не успеете отправить ответ в этот срок, ссылка станет недействительной и пользователь не получит результат.

{% endnote %}

### Статусы ответов

Ваш сервис должен корректно возвращать HTTP-коды статуса:

*   **200** — Сервис доступен и готов к работе (используется для проверки соединения).
*   **202** — Запрос принят и поставлен в очередь на обработку.
*   **503** — Сервис недоступен или перегружен.

{% note info "Успешный ответ" %}

Помимо кода `202`, в случае успешного приема задачи, обработчик обязательно должен вернуть тело ответа: `{"result": "OK"}`.

{% endnote %}

## Типы запросов

Структура входящего параметра `prompt` зависит от категории запроса (текст, аудио или изображение).

{% list tabs %}

- Текст (text)

    Для текстовых запросов (чат-боты, копирайтинг) в поле `prompt` передается **строка** с текущим запросом пользователя.

    Для формирования корректного запроса к GPT-подобным моделям, вам необходимо собрать массив сообщений, включающий:
    1.  **System (роль):** Берется из поля `payload_role` (инструкция для нейросети).
    2.  **Context (история):** Массив предыдущих сообщений из поля `context` (если параметр `collect_context` был активен).
    3.  **User (запрос):** Сам текст текущего запроса из поля `prompt`.

    **Пример сборки массива:**

    ```json
    [
        {
            "role": "system",
            "content": "$payload_role"
        },
        // ... элементы массива context ...
        {
            "role": "user",
            "content": "$prompt"
        }
    ]
    ```

- Аудио (audio)

    Для работы с аудио в поле `prompt` передается **объект**, содержащий информацию о файле и дополнительные параметры.

    Структура объекта `prompt`:
    *   **file** (`string`) — Ссылка на аудиофайл. **Важно:** Файл может прийти без расширения.
    *   **fields** (`object`) — Вспомогательная структура с техническими данными:
        *   **type** (`string`) — Content-type файла (например, `audio/ogg`). Это критически важно, если у файла отсутствует расширение.
        *   **prompt** (`string`) — Текстовый промпт (подсказка), который может помочь в распознавании (например, название компании или контекст разговора).

- Изображение (image)

    Для генерации изображений в поле `prompt` передается **объект (массив)** с параметрами генерации.

    Структура объекта `prompt`:
    *   **prompt** (`string`) — Текстовое описание того, что нужно изобразить.
    *   **style** (`string`) — Стиль генерации (например, "Advertising poster style, professional").
    *   **format** (`string`) — Формат изображения (например, "square", "landscape").

{% endlist %}

## Входные параметры

Ниже приведено описание всех полей, которые приходят в теле POST-запроса к вашему сервису.

#||
|| **Параметр** | **Тип** | **Описание** ||
|| **category** | `string` | Тип запроса. Возможные значения: `text`, `audio`, `image`. ||
|| **prompt** | `any` | Основное содержимое запроса.
Структура зависит от `category` (см. раздел "Типы запросов" выше). ||
|| **callbackUrl** | `string` | Адрес, на который ваш сервис должен отправить результат после генерации. ||
|| **errorCallbackUrl** | `string` | Адрес для отправки информации об ошибке, если генерация не удалась. ||
|| **ttl** | `integer` | Время жизни (в секундах) ссылки `callbackUrl`. По истечении этого времени ссылка станет недействительной. ||
|| **auth** | `object` | Данные для авторизации. ||
|| **payload_raw** | `string` | Сырое значение запроса пользователя (без препромптов и ролей). При использовании Копилота здесь будет символьный код использованного промпта. ||
|| **payload_provider** | `string` | Символьный код провайдера препромпта. В контексте Копилота здесь будет `prompt`. ||
|| **payload_prompt_text** | `string` | Если `payload_provider = prompt`, содержит сырую текстовую инструкцию препромпта. Предназначено для самостоятельного анализа вашим сервисом. Подробнее в документации по [промптам](./prompts/markers.md). ||
|| **payload_markers** | `object` | Массив дополнительных маркеров, использованных при формировании запроса (например, `original_message`, `user_message`, `language`). Подробнее в документации по [промптам](./prompts/markers.md). ||
|| **payload_role** | `string` | Роль (системная инструкция), использованная при формировании промпта. Для GPT-моделей это должно уходить в поле `system` массива сообщений. ||
|| **context** | `array` | Массив предшествующих сообщений (история диалога) в хронологическом порядке. Например, список комментариев к посту. Первым в таком списке контекста считается авторское сообщение (сам пост). Важно: Объем контекста, отправляемого вашему провайдеру, зависит от указанного вами объема провайдера и типа подсчета (подробнее в документации по провайдеру). По умолчанию метод подсчета "токены", объем 16K. Отправлять контекст непосредственной нейронке вы должны только если передан параметр collect_context, равный true (1). В остальных случаях он передается как доп.информация на ваше усмотрение. ||
|| **max_tokens** | `integer` | Максимальное количество лексем (токенов) в ответе. Контролирует длину генерируемого текста. Необязательный параметр. ||
|| **temperature** | `double` | Температура генерации. Параметр контролирует случайность ответа. Низкие значения делают ответ более строго определенным, высокие — более творческим. ||
|#
