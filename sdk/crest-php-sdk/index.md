# CRest PHP SDK: установка и первый вызов

{% note tip "" %}

Выберите инструмент для разработки с AI-агентом:

- используйте [Битрикс24 Вайбкод](../../ai-tools/vibecode.md), чтобы создать приложение для Битрикс24 по описанию задачи без знания языков программирования. Агент напишет код и разместит приложение на сервере без ручной настройки хостинга
- используйте [MCP-сервер](../../ai-tools/mcp.md), чтобы разрабатывать интеграцию через REST API в своем проекте. Агент будет обращаться к официальной REST-документации

{% endnote %}

CRest PHP SDK — набор PHP-файлов для вызова методов Битрикс24 из серверного приложения. Файлы размещают на своем веб-сервере, в коде подключают `crest.php` и вызывают методы через `CRest::call`.

Чтобы CRest мог обращаться к Битрикс24, выберите сценарий подключения: входящий вебхук, локальное или тиражное приложение. Таблица ниже поможет выбрать подходящий вариант.

| Если нужно                                              | Выберите вариант                                |
|---------------------------------------------------------|-------------------------------------------------|
| Настроить внутреннюю интеграцию без создания приложения | [Входящий вебхук](#incoming-webhook)            |
| Создать приложение для одного Битрикс24                 | [Локальное приложение](#local-app)              |
| Устанавливать приложение в разных Битрикс24             | [Тиражное приложение](#market-app)              |

## Как начать работу

Сначала подготовьте сервер, затем настройте один вариант подключения из таблицы и выполните общий первый вызов.

### Подготовьте сервер

1. Скачайте [файлы CRest](https://github.com/bitrix-tools/crest)
2. Разместите на своем веб-сервере содержимое папки `src` — в ней лежат все файлы CRest
3. Убедитесь, что на сервере доступен модуль [cURL](https://www.php.net/manual/ru/book.curl.php) и установлен действующий SSL-сертификат
4. Откройте в браузере страницу `checkserver.php` по адресу вашего сайта, например `https://your-domain.ru/checkserver.php`, — скрипт проверит, что модуль cURL доступен и CRest может сохранять свои файлы

### Настройте входящий вебхук {#incoming-webhook}

Входящий вебхук — это ссылка с ключом доступа, по которой можно вызывать методы Битрикс24. Вебхук подходит для внутренней интеграции без приложения.

Создайте [входящий вебхук](../../local-integrations/local-webhooks.md) в Битрикс24 и выберите для него права доступа — для примера из этой статьи понадобится CRM. Скопируйте полный URL вебхука в константу `C_REST_WEB_HOOK_URL` файла `settings.php`:

```php
<?php

define(
    'C_REST_WEB_HOOK_URL',
    'https://example.bitrix24.ru/rest/1/your-webhook-code/'
);
```

### Настройте локальное приложение {#local-app}

Локальное приложение работает в одном Битрикс24 и использует протокол OAuth 2.0. При установке приложения Битрикс24 выдает ему токены — ключи доступа с ограниченным сроком действия. CRest продлевает их автоматически.

Выберите [сценарий установки локального приложения](../../settings/app-installation/local-apps/index.md) и заполните карточку приложения. Если у приложения есть основная страница, укажите ее URL. В поле «Путь для первоначальной установки» укажите URL файла `install.php` — этот файл из поставки CRest получает токены при установке приложения и сохраняет их. Скопируйте `client_id` и `client_secret` из карточки в файл `settings.php`:

```php
<?php

define('C_REST_CLIENT_ID', 'your-client-id');
define('C_REST_CLIENT_SECRET', 'your-client-secret');
```

Если вы вписали `client_id` и `client_secret` в `settings.php` уже после установки приложения, переустановите его — тогда `install.php` запустится повторно и сохранит токены.

### Настройте тиражное приложение {#market-app}

Тиражное приложение предназначено для установки в разных Битрикс24 и также использует OAuth 2.0. Создайте приложение в партнерском кабинете и сохраните его, чтобы получить `client_id` и `client_secret`. Укажите эти значения в файле `settings.php`:

```php
<?php

define('C_REST_CLIENT_ID', 'your-client-id');
define('C_REST_CLIENT_SECRET', 'your-client-secret');
```

Выберите [сценарий установки тиражного приложения](../../settings/app-installation/mass-market-apps/index.md). От выбранного сценария зависит, в какое поле карточки версии вписать URL файла `install.php`:

- у приложения есть интерфейс — выберите сценарий с мастером установки и укажите URL файла `install.php` в поле «Ссылка на установочное приложение»
- у приложения нет интерфейса — выберите сценарий с callback-обработчиком и укажите URL файла `install.php` в качестве обработчика. Битрикс24 вызовет его автоматически при установке

Если у приложения есть интерфейс, укажите в карточке версии URL его основной страницы. После сохранения версии установите приложение на доступный вам Битрикс24 и проверьте его работу.

### Выполните первый вызов

После настройки авторизации откройте файл `index.php` — он входит в поставку CRest и лежит рядом с `crest.php` и `settings.php`. Замените его содержимое кодом первого вызова, а если файла нет — создайте его.

Код подключает `crest.php` и вызывает метод через `CRest::call`:

```php
<?php

require_once __DIR__ . '/crest.php';

$result = CRest::call(
    'crm.item.add',
    [
        'entityTypeId' => 3,
        'fields' => [
            'name' => 'Иван',
            'lastName' => 'Иванов',
        ],
    ]
);

if (isset($result['error']))
{
    print_r($result);
}
else
{
    echo 'Создан контакт с идентификатором ' . $result['result']['item']['id'];
}
```

Чтобы выполнить вызов, откройте `index.php` в браузере по адресу вашего сайта, например `https://your-domain.ru/index.php`.

Метод [`crm.item.add`](../../api-reference/crm/universal/crm-item-add.md) создает элемент CRM. Тип элемента задает параметр `entityTypeId`: значение `3` — это контакт. В ответе метод возвращает созданный элемент, поэтому идентификатор нового контакта придет в `$result['result']['item']['id']`. Если вместо этого пришла ошибка, сначала проверьте права: у вебхука или приложения должен быть доступ к CRM.

## Связь с другими инструментами

**Контекст пользователя.** Способ авторизации определяет, от чьего имени CRest выполняет запросы. Входящий вебхук выполняет запросы от имени пользователя, который создал вебхук. Локальное и тиражное приложения по умолчанию используют токены пользователя, который установил приложение. Если нужно выполнять запросы от имени пользователя, который открыл приложение, настройте [работу в контексте текущего пользователя](./using-in-users-context.md).

**Другие SDK.** Сравнить CRest с другими библиотеками можно в [обзоре SDK](../index.md). Для PHP также доступен [B24PhpSDK](../b24phpsdk/index.md).

## Что важно учитывать

- Не публикуйте URL входящего вебхука, `client_secret` и токены авторизации
- Доступ к данным зависит от прав пользователя, от имени которого выполняется запрос. Кроме того, каждому методу нужен свой scope — разрешение на работу с определенным разделом Битрикс24. Значения scope смотрите в справочнике [Права доступа](../../api-reference/scopes/permissions.md)
- Если кодировка проекта отличается от UTF-8, может потребоваться изменить кодировку файлов SDK и объявить в `settings.php` константу `C_REST_CURRENT_ENCODING`, например `define('C_REST_CURRENT_ENCODING', 'windows-1251');`
- Для тиражного приложения на нескольких Битрикс24 предусмотрите собственное хранение токенов. Для этого переопределите методы `getSettingData` и `setSettingData`: базовый вариант хранит токены одного Битрикс24 в файле `settings.json` и для такого сценария не предназначен. Пример наследования класса CRest с переопределением метода есть на странице [работы в контексте текущего пользователя](./using-in-users-context.md)
