# Как вызывать методы REST API

{% note tip "" %}

Если вы разрабатываете интеграции для Битрикс24 с помощью AI-инструментов (Codex, Claude Code, Cursor), подключите [MCP-сервер](../../sdk/mcp.md), чтобы ассистент использовал официальную REST-документацию.

{% endnote %}

Методы REST API Битрикс24 вызывают по HTTP. В запросе передают авторизацию и параметры, а в ответ получают `JSON` или `XML`.

Если вы делаете первый вызов, начните с авторизации и общей схемы запроса. Если интеграция уже работает, переходите сразу к материалам про кодирование, `batch` или списочные методы.

> Быстрый переход: [все материалы раздела](#all-materials)

## Что подготовить перед первым вызовом

**Адрес Битрикс24.** Подготовьте домен, к которому отправляете запросы. Если Битрикс24 меняет адрес, обработайте редирект по рекомендациям из статьи [об изменении адреса Битрикс24](./change-domen.md).

**Название метода и параметры.** Подготовьте имя метода и параметры запроса. Для первого теста возьмите пример со страницы [Первый вызов REST API](../../first-steps/first-rest-api-call.md) или простой метод чтения, например `user.current`.

## Как выбрать способ авторизации

|| **Сценарий** | **Что выбрать** ||
|| Быстрый первый вызов, проверка метода или локальная интеграция без приложения | Входящий вебхук ||
|| Локальное или тиражное приложение, работа от имени разных пользователей, управление доступом | OAuth 2.0 ||

Подробности смотрите в статье [Авторизация в REST](./authorization.md).

## Как выполнить первый вызов {#first-call}

1. Выберите способ авторизации: вебхук или OAuth 2.0
2. Соберите URL запроса по схеме из статьи [Как выполняется запрос](./general-principles.md)
3. Передайте параметры в `GET` или `POST`. Для методов с поддержкой `JSON` используйте `POST` с телом запроса в этом формате
4. Закодируйте специальные символы и сложные параметры по правилам из статьи [Кодирование данных](./data-encoding.md)
5. Выполните запрос и проверьте поле `result` или описание ошибки в ответе
6. Если метод возвращает список, обработайте поля `total` и `next` по статье [Особенности списочных методов](./list-methods-pecularities.md)
7. Если нужно сократить число обращений, объедините вызовы через [batch](./batch.md)

## Что учитывать в рабочих интеграциях

**Приложение.** Если интеграция работает как приложение, способ получения токена и сценарий работы с ним зависят от установки и контекста приложения. Подробности смотрите в разделе [OAuth 2.0](../oauth/index.md).

**Кодирование.** Проверяйте URL-кодирование при передаче специальных символов, например `&`, `?`, `%`, `[`, `]` и `#`. Отдельно проверьте вложенные запросы в [batch](./batch.md).

**Формат тела запроса.** Для массивов и вложенных объектов передавайте `POST` с `Content-Type: application/json`. Так структура параметров сохраняется без преобразования в строку запроса.

**Пакетные вызовы.** Метод [batch](./batch.md) выполняет до 50 команд за один запрос и позволяет использовать результат предыдущей команды в следующей.

**Лимиты.** При частых и тяжелых вызовах учитывайте [лимиты REST API](../performance/limits.md). Это особенно важно для списков и пакетных запросов.
